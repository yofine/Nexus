## 竞品优劣势分析

### Nexus vs Claude Squad（最强竞品）

**Claude Squad 的优势**
* 已经过真实用户验证，GitHub 有实际用户基础
* git worktree 强隔离方案从根本上解决了文件冲突问题，比"信息共享让 Agent 自己协调"更可靠
* TUI 比 Web 启动更快，无浏览器依赖，对开发者更自然
* 依赖 tmux 意味着天然的 session 持久化，断网重连零成本

**Claude Squad 的劣势**
* 纯 TUI 信息密度有限，FileTree / Diff / CR 这类视图在终端里做体验很差
* git worktree 强隔离导致 Agent 之间完全割裂，无法感知彼此，合并时冲突反而后置了
* 没有任务分发和广播概念，批量下发任务需要手动逐个操作
* 不跟随 repo，workspace 配置无法版本化

---

### 核心优劣势矩阵

| 维度 | Nexus 优势 | Nexus 劣势 / 风险 |
|---|---|---|
| **界面** | Web 界面信息密度高，FileTree + Diff + Terminal 同屏 | 比 TUI 重，冷启动需要开浏览器 |
| **协作模型** | 共享 repo + agents.yaml 信息共享，Agent 感知彼此 | 依赖 Agent 自己协调，不如 worktree 硬隔离可靠 |
| **工作流** | CR → 直接发回 Agent 形成闭环，广播批量分发 | 功能多意味着实现复杂度高 |
| **配置** | .nexus/ 跟随 repo，可版本化，tmuxinator 式体验 | 比竞品配置更复杂 |
| **定位** | 唯一以 Repo 为中心的 Web 工作台 | 赛道新，用户需要培养习惯 |

## 现实世界需求可行性分析

### 有利因素

**需求侧正在爆发**

Claude Code、OpenCode、Qwen-Code 这批工具在 2025 年集中成熟，开发者同时跑多个 Agent 的场景从少数极客行为变成了主流工作方式。Claude Squad 的出现本身就是需求验证，而它的 TUI 形态留下了明显的体验空白。

**Web 工作台有天然优势**

Diff 查看、文件树、代码 Review 这类操作在终端里体验本来就差，Web 是正确的载体。VSCode 和 GitHub 已经教育了开发者用 Web/GUI 做 Review，迁移成本低。

**技术栈成熟**

node-pty、xterm.js、chokidar、simple-git 都是经过大量生产验证的库，没有技术风险，核心难点在于工程整合而非技术探索。

---

### 风险因素

**风险 1：Agent CLI 本身在快速演化**

Claude Code 已经原生支持 Agent Teams + split pane 模式，官方在往"多 Agent 协作"方向走。如果 Claude Code 官方做了更好的多 Agent 管理界面，Nexus 对 claudecode 用户的价值会被压缩。

* 应对：Nexus 的价值在于**异构多 Agent**（同时跑 claudecode + opencode + kimi-cli），这是官方工具不会做的方向

**风险 2：共享 repo 的协调模型可能在实践中不够可靠**

"让 Agent 自己通过 agents.yaml 感知彼此然后协调"这个假设依赖 Agent 的指令遵循能力。实际上 Agent 可能忽略上下文信息，仍然产生冲突。

* 应对：这是设计上需要诚实面对的取舍。可以在 P2 阶段加入**文件级冲突检测警告**作为补充，不做硬锁但至少给用户可见的提示

**风险 3：单人本地工具的商业化路径窄**

定位是本地单机工具，天然没有多人协作、云端同步等付费点，开源后很难变现。

* 应对：如果目标是个人效率工具或开源项目，这不是问题；如果有商业化预期，需要提前想清楚

**风险 4：维护多个 Agent CLI 的适配成本**

claudecode / opencode / kimi-cli / qwencode 各自的 CLI 接口、statusline 支持、continue 参数都不一样，而且会持续更新。claudecode 有 statusline JSON API，其他的没有，导致功能不对等。

* 应对：设计一个 Agent Adapter 层，每个 Agent 有自己的适配器，核心功能（PTY 管理）与 Agent 特性（statusline）解耦，缺失的特性优雅降级

## 综合判断

```
需求真实性    ████████░░  强，有竞品验证，有体验空白
技术可行性    █████████░  高，无技术风险，工程复杂度可控
差异化空间    ███████░░░  中，Web + Repo 中心是空白，但官方工具在追赶
实现风险      ████░░░░░░  主要风险在协调模型假设和多 CLI 适配维护
```

**最现实的路径建议：**

P0/P1 阶段专注做好**单一 claudecode 的多实例管理 + Web 工作台**，这个场景需求最强、用户最多、技术最成熟。claudecode statusline API 能拿到最丰富的元数据，体验会最好。异构多 Agent 支持作为后续扩展，而不是一开始就铺开适配四个 CLI 的成本。
