# 活动面板（Activity Panel）设计文档

> 活动面板是 Nexus 的核心差异化功能，目标是帮助用户在多 Agent 并行工作时提升效率和质量。

## 实施状态

| 视图 | 状态 | 说明 |
|------|------|------|
| Agent 仪表盘 | ✅ 已完成 | `AgentDashboard.tsx` — 卡片式 dashboard，顶部色条标识 |
| Modules | 🔲 待实现 | 需后端模块识别 API，当前仍为原 Directory 视图 |
| Imports | ✅ 已改造 | 聚焦活跃文件+邻居节点，新增上游变更警告，精简节点信息 |
| Conflicts | ✅ 已完成 | `ConflictsPanel.tsx` — 直接/依赖/历史三类冲突检测 |
| Files | ✅ 已完成 | `FilesPanel.tsx` — 可排序文件表格，变更量/agent归属/冲突标记 |
| ~~Review(Heatmap)~~ | ❌ 已移除 | 与 Review Tab 职责冲突，Review 优先级能力归入 Review Tab |
| ~~Temporal~~ | ❌ 已移除 | 从 ViewMode 中删除，职责合并到 Timeline |
| Timeline 增强 | ✅ 已完成 | `TimelineSwimlane.tsx` — 新增泳道模式，列表/泳道可切换 |
| Active Agents | ✅ 已微调 | 文件路径改为最后两级，新增操作类型图标 |

## 设计原则

- 每个视图必须有**明确的使用场景**，而非堆砌信息
- 每个 Agent 实例有**专属颜色**（基于 pane 索引，非 agent 类型），贯穿所有视图
- 信息层级清晰：关键信息一眼可见，细节按需展开

## 视图模式

### 1. Imports（依赖关系视图）

**设计目标：冲突预警 + 影响链感知**

用户在此视图中应能快速回答：
- 哪些 Agent 正在修改有依赖关系的文件？（潜在冲突）
- 某个文件被修改后，影响链有多长？

**当前实现问题（v1 Review）：**

| 问题 | 说明 |
|------|------|
| 信息过载 | 节点塞了 role 标签、git 状态、热度点、agent 名称、fan in/out 条。缩放后 8px 文字不可读，关键信息被淹没 |
| 冲突发现不够直觉 | 仅标记直接冲突（同文件多 agent 写入），未沿依赖链传播警示。核心场景"A 改了 X，B 在改依赖 X 的 Y"未被覆盖 |
| 静态全量布局 | dagre 平铺所有文件，活跃节点散落在大量灰色无关文件中，难以聚焦 |
| 连线噪音 | 所有 import 关系都画出来，线多时成为视觉噪音。有价值的是"活跃文件的影响链"而非完整依赖图 |

**改进方向：**

- 默认只展示被 agent 触及的文件 + 它们的直接依赖/被依赖节点（可切换全量）
- 沿依赖链传播冲突标记：A 改了 X → 依赖 X 的文件也高亮提示"上游被修改"
- 节点信息精简，突出两件事：谁在操作、是否有冲突风险
- 非活跃连线大幅弱化或隐藏，活跃影响链用 agent 颜色强调

### 2. Modules（模块视图）— 原 Directory，重新定位

**设计目标：宏观态势感知 — 各 Agent 的工作分区是否合理**

与 Imports 视图的关系：**不同缩放级别**
- 模块视图是**战略地图**：一眼看到"Agent A 在 PTY 管理模块，Agent B 在前端 UI 模块"
- Imports 视图是**战术地图**：具体文件的影响链和冲突细节
- 使用流程：模块视图发现分区异常 → drill down 到 Imports 看具体文件

用户在此视图中应能快速回答：
- 各 Agent 分别在哪个功能区工作？
- 有没有两个 Agent 挤在同一个模块里？（需要协调）
- 模块之间的依赖关系如何？（跨模块修改的风险评估）

**v1 问题（原 Directory 视图）：**

| 问题 | 说明 |
|------|------|
| 粒度不对 | 目录 ≠ 模块。目录是物理路径，模块是功能边界 |
| 缺乏语义 | `src/pty/` 显示为目录名，用户需要自己理解这是"PTY 管理" |
| 与 Imports 重叠 | 冲突检测、agent 标签等信息在 Imports 已有，只是换了维度 |
| 无法 drill down | 点击目录无法展开查看具体文件 |

**新设计：自动模块识别 + 模块级拓扑**

示例效果（以 Nexus 自身为例）：
```
[PTY 管理] ←→ [状态中心] ←→ [配置系统]
                 ↕
           [WebSocket 层] ←→ [前端 UI]
                 ↕
            [文件监听]  [Git 服务]
```
每个模块节点用 Agent 颜色标记谁在操作，模块间连线表示依赖。

**实现路径：**

Phase 1 — 模块自动识别（后端 `/api/modules`）
1. 基于已有的 `/api/deps` 依赖图数据
2. 识别策略（组合使用）：
   - **目录聚合**：同目录下的文件默认归为一个模块，目录名作为模块候选名
   - **入口文件检测**：有 `index.ts` 的目录视为一个模块，入口文件的导出定义模块边界
   - **import 聚类**：互相 import 密集但对外 import 稀疏的文件组自动聚合（社区检测算法思路）
   - **命名启发**：目录名到语义标签的映射（`pty/` → "PTY 管理"，`ws/` → "WebSocket 层"）
3. 输出：`{ modules: [{ id, label, files[], deps[] }] }`

Phase 2 — 前端模块视图渲染
1. 模块节点替代目录节点，展示：模块名、文件数、活跃 Agent 标签
2. 模块间连线 = 跨模块 import 关系聚合（线粗细 = 耦合强度）
3. 点击模块可展开文件列表或跳转 Imports 视图（filtered）

Phase 3 — 用户可配置（可选）
- `.nexus/modules.yaml` 允许用户手动定义/覆盖模块划分
- 自动识别结果作为默认值，用户可调整

### 3. ~~Heatmap / Review~~ → 已移除，替换为 Files

**移除原因：**

原 Heatmap 视图重新定位为 "Review Priority" 后，发现与已有的 **Review Tab**（`GitDiffPanel.tsx`）职责重叠：
- Review Tab 已有完整的 staged/unstaged diff 浏览、行级 comment、commit/push 等功能
- Review 优先级排序（哪些文件该优先看）更适合作为 Review Tab 的增强，而非独立面板
- 活动面板应聚焦于**运行时态势感知**，而非收尾阶段的 CR

**替代方案 — Files 面板：**

新增 `FilesPanel.tsx` 作为第 5 个面板，提供**文件维度的全局概览**：
- 可排序表格：文件名、变更量（+N/-N 可视化条）、操作次数、Agent 归属、最后活动时间
- 冲突文件高亮标记
- 点击直接打开文件查看器
- 底部汇总：总文件数、总增删行数、冲突数

与其他视图的关系：
- **Imports** 看依赖链关系（图），**Files** 看扁平列表（表格）
- Files 面板是快速定位"谁改了什么"的工具，比 Imports 更直接
- Review 优先级能力（按变更量排序、多 agent 交叉标记）自然内置在排序逻辑中

### 4. Agent（Agent 产出仪表盘）— 重新定位

**设计目标：评估每个 AI Agent 的产出和状态 — "AI 员工绩效面板"**

用户在此视图中应能快速回答：
- 每个 Agent 产出了多少？（变更行数，而非操作频次）
- 哪个 Agent 卡住了？（长时间无产出 / context 接近满）
- 任务分配是否均衡？（一个改了 30 文件，另一个只改了 3 个）
- 各 Agent 的资源消耗如何？（费用、context 使用率）

**v1 问题：**

| 问题 | 说明 |
|------|------|
| 只是分组文件列表 | 按 Agent 分组展示文件卡片，没有"评估"维度 |
| 用了拓扑图形式 | Agent 绩效是表格/卡片问题，不是图论问题，拓扑图形式不合适 |
| 文件卡片信息过重 | 复用 FileNodeComponent，在此场景下大部分信息无意义 |
| 缺少关键指标 | 没有变更行数、费用、context%、工作节奏等管理视角的数据 |

**改进方向：**

改为 Agent 卡片/dashboard 形式，每个 Agent 一张卡片，展示：

```
┌─ Agent A (Claude Code) ─────────────────────┐
│  +234 / -56 lines  │  12 files  │  $0.42    │
│  Context: ████████░░ 78%                     │
│  Status: running  │  uptime: 23m            │
│                                              │
│  活跃文件: store.ts, Layout.tsx, ...         │
│  节奏: ▂▃▅▇█▇▅▃ (最近 30min 活动曲线)       │
└──────────────────────────────────────────────┘
```

关键指标：
- **改动量**：增/删行数（git diff 统计）
- **文件数**：touch 过的文件总数
- **资源消耗**：费用 (costUsd)、context 使用率 (contextUsedPct)
- **工作节奏**：迷你时间线/活动火花图，反映是持续产出还是卡住
- **质量信号**：冲突文件数、改了又改回的文件数（churn）
- **状态**：running/waiting/idle + 持续时间

### 5. ~~Temporal（时序视图）~~ → 砍掉，合并到 Timeline 增强

**结论：不再作为独立视图，时序需求由底部 Timeline 增强覆盖（见公共组件 → Timeline 部分）**

v1 问题：
- 时间轴不可见（网格按序号排列，看不出时间间隔）
- 只记录 first-touch，后续反复操作不体现
- 连线是单链序列，丢失了多 Agent 并行性
- 与底部 Timeline 高度重叠

## 公共组件

### Active Agents 状态条（顶部）

**设计目标：实时显示每个 Agent 此刻在干什么**

**v1 评估：定位清晰，基本可用，微调即可**

当前实现：有 agent 操作文件时出现，每个 agent 一个 chip（呼吸圆点 + 名称 + 当前文件名）。

小改进：
- 文件名只显示 `split('/').pop()`，同名文件（多目录下的 `index.ts`）无法区分 → 显示最后两级路径
- 缺少操作类型 → 展示 `current.action`（read/edit/write 图标）

### Timeline（底部）— 需增强，承接 Temporal 视图的职责

**设计目标：多 Agent 工作的时序回溯 + 并行可视化**

**v1 评估：**

当前实现做了基础功能：
- 按时间倒序列出操作条目（agent 名 + 操作图标 + 文件名 + 相对时间）
- 支持按 Agent 筛选（filter chips）
- 新条目有入场动画
- 上限 30 条

问题：
| 问题 | 说明 |
|------|------|
| 纯列表，无并行感知 | 所有 Agent 操作混在一条流里，看不出"A 和 B 同时在工作"还是"A 做完 B 才开始" |
| 无时间刻度 | 只有相对时间（"3s ago"），无法感知操作间隔和节奏 |
| 上限 30 条 | 多 Agent 高频操作时很快就刷没了，无法回溯 |
| 不可交互 | 不能点击条目跳转到对应文件的 diff |

**增强方向（承接原 Temporal 视图的职责）：**

两种展示模式切换：
1. **列表模式**（当前）— 保留，适合快速浏览最新动态
2. **泳道模式**（新增）— 每个 Agent 一条水平泳道，横轴是时间，操作事件标记在对应时间点上
   - 一眼看出各 Agent 的并行性和工作节奏
   - 同一文件被不同 Agent 操作时用连线标出
   - 可缩放时间范围（最近 5min / 30min / 1h）
   - 点击事件可跳转 diff

### Conflicts（冲突中心）— 新增面板

**设计目标：集中展示和处理所有冲突，提供可操作的响应**

当前冲突信息分散在 Imports/Modules 各视图中，用户需要自己在拓扑图里找黄色节点。
Conflicts 面板将冲突提升为一等公民，以列表形式直接呈现，带操作按钮。

**冲突类型：**
1. **直接冲突** — 多个 Agent 同时写入同一文件
2. **依赖冲突** — Agent A 修改了文件 X，Agent B 在修改依赖 X 的文件 Y（上游变更风险）
3. **模块冲突** — 多个 Agent 在同一模块内工作（Modules 视图检测到的重叠）

**交互：**
- 每条冲突展示：涉及文件、涉及 Agent（pane 颜色标识）、冲突类型、严重程度
- 可操作：查看 diff、跳转到 Imports 视图聚焦该冲突链、（未来）通知 Agent 暂停

## 视图顺序

按使用频率和工作流阶段排列：

| 顺序 | 视图 | 对应阶段 | 理由 |
|------|------|----------|------|
| 1 | **Agent** | 全程常驻 | 最高频，管理者主屏：各 Agent 状态、产出、是否卡住 |
| 2 | **Modules** | 启动/分工 | 确认工作领地划分是否合理 |
| 3 | **Imports** | 监控/排查 | 发现重叠后 drill down 看文件级冲突链 |
| 4 | **Conflicts** | 监控/响应 | 冲突发生时集中处理（可考虑有冲突时自动提示切换） |
| 5 | **Files** | 全程 | 快速查看谁改了什么文件、变更量多少，扁平表格适合快速定位 |

底部 Timeline 始终可见，不占 tab 位。

## Pane 颜色体系

每个 Agent pane 实例按创建顺序分配专属颜色（12 色循环），用于在所有视图中一致地标识该 Agent 的行为：

```
#7C6AF7 purple    #3B82F6 blue      #10B981 emerald
#F59E0B amber     #EF4444 red       #EC4899 pink
#06B6D4 cyan      #F97316 orange    #8B5CF6 violet
#14B8A6 teal      #84CC16 lime      #E879F9 fuchsia
```

颜色出现位置：
- AgentPane header 中的圆点标识
- 活动面板顶部 agent 列表
- 拓扑图中的节点边框、光晕、agent 标签
- Timeline 中的竖条和名称
- 依赖连线着色
- MiniMap 节点着色

> 注：ReplayViewer（历史回放）仍使用 agent 类型品牌色（如 Claude Code = #D97757），因为回放时原始 pane 可能已不存在。
