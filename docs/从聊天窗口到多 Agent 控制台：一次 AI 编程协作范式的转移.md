过去一段时间，随着工作中 AI Coding 的占比越来越高，我明显地感觉到：AI IDE 正在变强，但它们默认的单 Agent 协作方式，已经逐渐不适合我了。

今天主流的 AI 开发体验，仍然是人与一个 Agent 围绕同一个任务持续协作。但在真实工作里，我真正需要的是：把复杂需求拆成多个子任务，交给不同 Agent 并行执行，而我只在关键节点做 ⁠Review，并对最终结果进行验收和整合。

所以我想讨论的是 AI 编程协作范式的转移：主流 AI IDE 优化的，仍然是“人与一个 Agent 持续协作”；而我真正需要的，则是把复杂需求拆解后交给多个 Agent 并行执行，自己把工作重心转向编排、⁠Review 和最终验收。

## 当单 Agent 协作开始不够用
问题不是 AI IDE 不够强，而是它默认的协作节奏太像单线程。

在主流 AI IDE 里，我通常是在和一个 Agent 持续协作：我提需求，它读取上下文、改动多个文件、执行命令，再返回结果，等我确认后进入下一轮。这种方式当然有效，但它也意味着，我的大量时间都消耗在“等它返回、看它改了什么、决定要不要继续”这条链路里。

真正浪费时间的，很多时候不是 AI 写得不够快，而是整个过程要求我不断等待、阅读、确认、继续推动。只要任务还在这条链上推进，人就没有真正从执行流程里抽离出来。

但现在，我更常见的工作流已经变了。比起让一个 Agent 长时间陪我在同一个工作区里来回推进，我更希望把复杂需求先拆开，让不同 Agent 并行执行；而我自己则把时间用在继续分发任务、Review 已有代码、检查 diff，以及对最终结果做验收和整合。
也就是说，我不是不再需要 AI，而是越来越少把“和一个 Agent 持续结对”当成默认方式。 


我更需要的是：当一个 Agent 在执行时，我可以继续推进别的工作，让人和 AI 真正并行起来。

## 我在找一种新的协作方式
当我开始明确地需要这种工作方式之后，我其实不是立刻想到自己做工具，而是先去看现有方案里有没有已经很接近的。

我想找的东西很明确：
它不是单纯的聊天窗口
它能承接多个 Agent 并行工作
它最好把 Review 放回工作流中心
它应该更像一个工作台，而不是一个对话框

也就是从这时候开始，我关注的重点不再只是“哪个 Agent 更强”，而是：有没有一种更适合新协作方式的交互界面。

## OpenCode 的 Web 模式给了我最接近的雏形
在我用过的这些工具里，OpenCode 的 Web 模式最接近我想要的那种工作方式。

它让我看到了一种更适合我的交互方式：Agent 先执行，我在关键节点看改动、给反馈、做验收，而不是一直围着一个聊天窗口来回推进。

这和我自己的使用变化很接近。比起和一个 Agent 长时间持续对话，我更希望它先跑一段时间，我再回来审查结果、决定下一步。
但它也让我意识到还差什么：一是对多 Agent 并行的体现还不够强，二是它本质上还是单 Agent 工作流。而我真正想要的，是把复杂需求拆开，交给多个 Agent 同时推进，再由我统一分配、Review 和收束。

也就是从这里开始，我才真正动了自己做一个工具的念头。

## Nexus：我对新范式的一次交互设计实践
Nexus 的定位很简单：一个 CLI AI Agent 多实例管理的 Web UI。

我做它，不是为了再做一个 AI IDE，而是为了给另一种工作方式提供界面：一个人同时管理多个并行工作的 Agent。
它可以跑在本地，也可以部署在服务端；但不管在哪里运行，它解决的都是同一个问题：当你开始同时使用多个 Agent 时，怎么把它们放进一个顺手、可管理、可审查的工作界面里。

所以 Nexus 从一开始就不是按聊天窗口去设计的，而是按控制台去设计的。左边管 Pane，中间放多个 Agent 工作区，右边看 diff、Review 和文件。

在这种模式里，我最需要的不是聊天，而是随时知道：
哪些 Agent 在跑
它们各自负责什么
当前做到哪一步
哪些结果该我来 Review

在 Nexus 里，每个 Pane 都是一个独立的 Agent 工作单元。它有自己的任务、目录、状态和历史，折叠起来就像一张任务卡片。这样我能很快判断该看谁、该审谁、该继续把什么任务分发出去。

我不需要一个更大的聊天框，我需要的是一个能同时看多个执行单元、能随时插入 Review、也能继续分发任务的工作台。

## Nexus 想解决什么问题
Nexus 真正要解决的，其实就几件事。
1. 把单 Agent 的串行协作，变成多 Agent 的并行执行
在 Nexus 里，我可以在一个 Web UI 里同时运行多个 CLI Agent。每个 Agent 都有自己的 Pane、自己的任务、自己的工作目录。
这带来的变化不是“多开几个窗口”，而是工作方式变了：原来一个任务只能围着一个 Agent 一轮轮推进，现在可以把不同子任务同时分出去，让多个执行单元并行工作。
2. 让任务拆分真正落到代码库里
Nexus 里每个 Pane 都可以绑定自己的 workdir 和任务说明，所以任务拆分不再只是脑中的计划，而是能直接落到仓库结构里的执行单元。
比如：
一个 Pane 处理 auth/
一个 Pane 处理测试
一个 Pane 处理文档
我自己负责看整体 diff 和最后收束
这样复杂需求就可以被拆开，而不是只能交给一个 Agent 串行推进。
3. 解决的不只是“人怎么管多个 Agent”，还有“多个 Agent 怎么协作”
Nexus 不是简单把多个终端摆到一个界面里，而是要让多个 Agent 真正进入同一个工作流。它们可以围绕同一个仓库协作，也可以在需要的时候切到隔离的 git worktree 里独立执行。
也就是说，Nexus 解决的不是单纯的多开，而是：
人怎么分配任务
人怎么判断该看谁、该审谁
Agent 和 Agent 怎么在同一个任务系统里配合，而不是彼此割裂
4. 把 Review 放回工作流中心
Nexus 不是想让你和 Agent 一直对话，它更像一个 Review 驱动的工作台。
右侧可以直接看 diff 和代码变化，我可以一边看结果，一边决定是继续、返工，还是收下。更重要的是，Review 不是停在“我看完了”这一步，而是能直接回到某个 Agent：我可以把反馈发回对应的 Pane，让它继续改。
所以 Nexus 要解决的，不只是“多个 Agent 同时跑”，而是这些并行执行出来的结果，怎么被人有效地审查、比较和收束。
5. 把不同 Agent 放进同一个控制台里统一管理
最后一点是，Nexus 不是只跑一种 Agent。不同 Pane 可以接不同的 CLI Agent，这样我不用在多个终端和多个窗口之间来回切换。
对我来说，这不是为了“多模型很酷”，而是因为现实里不同任务本来就适合不同的执行方式。Nexus 做的，是把这些执行单元放进同一个工作台里统一管理。
AI IDE 解决的是人与一个 Agent 如何协作，Nexus 更关注的是：当你开始同时使用多个 Agent 时，怎么把它们组织进一个可管理、可审查、可收束的工作界面里。

## 为什么还要重复造轮子
另一个很自然的问题是：既然已经有不少现成工具了，为什么我还要再做一个？

其实已经有不少工具在往多 Agent 方向走了，比如 Claude Squad、Agent Deck、Agents UI、Warp。这说明需求本身没问题：大家确实开始同时跑多个 Agent，把更多任务并行起来。

但这些工具解决的重点，和我想做的东西还是不太一样。

有的更偏终端里的多 session 管理，有的更像 AI agent 的统一控制台，有的把多 Agent 能力直接做进终端或桌面环境里。这些方向都对，也都各有价值。尤其像 Claude Squad 这种，把 git worktree 隔离和终端工作流结合起来，思路很扎实。
但对我来说，我真正想做的，是一个更适合多 Agent 工作方式的管理界面。我想要的是：
在一个界面里看到多个 Agent 的状态
同时看 repo、diff、文件树
Review 之后能直接把反馈送回某个 Agent
在同一个工作台里持续分配、审查和收束

而 Web UI 恰好有两个很实际的优势：一是门槛更低，二是交互能力更强。很多事情放在终端里能做，但不代表是最自然的体验；像文件树、diff、Review、评论回流这类操作，放在 Web UI 里会顺手很多。

换句话说，我想做的重点不是“让多个 Agent 跑起来”，而是：当我真的开始同时使用多个 Agent 之后，怎么把它们放进一个顺手、可管理、可审查的工作界面里。

## 为什么默认共享工作区，而不是默认 git worktree
Nexus 现在也支持 git worktree，但它是创建 Pane 时的可选模式，不是默认值。

原因很简单：对我来说，多 Agent 的默认场景首先是协作，不是隔离。

git worktree 很适合高风险重构、实验性改动，或者需要独立审查的任务。但如果一上来就把所有 Agent 都隔离开，其实等于默认它们彼此之间不需要配合，只是各自完成各自的事。

而我更想要的默认状态，是它们围绕同一个 repo、同一个目标一起往前推进。
共享工作区的好处也很直接：
改动会更早暴露
问题会更早发现
diff 可以持续看
人可以在执行过程中纠偏，而不是等到最后再统一收拾

所以 Nexus 的取舍不是反对隔离，而是把两种方式放到不同的位置上：
默认共享工作区，服务协作
按需启用 git worktree，服务隔离

默认共享工作区，不是因为我低估冲突，而是因为我希望把冲突发现和质量纠偏前移到执行过程中，而不是后置到合并阶段。
换句话说，共享和隔离不是对立选项，而是同一套多 Agent 工作流里的两种执行策略。

## 重塑人与 AI 的生产关系
这轮 AI 编程范式变化，背后有两个很明显的推动因素。

第一是 Agent 的成熟。不管是 Claude Code 在编程场景里的能力，还是 OpenClaw 在通用任务上的表现，都在不断抬高大家对 Agent 能力边界的判断。
第二是 模型的进步。很多人都觉得 Claude Opus 4.6 的提升非常明显，我自己的实际使用感受也是这样。模型和 Agent 这两方面的进步放在一起，说明 AI 已经越来越适合承担更完整、更长链路的任务。

工具越来越强之后，真正需要提升的，反而是人使用工具的方式。

过去那种“人提一句，AI 回一句，再逐轮确认”的协作方式当然还有效，但它默认 AI 更像助手，需要被持续牵着往前走。可当工具能力已经发展到今天这个阶段，真正开始限制生产力的，往往不再是模型本身，而是我们还在沿用旧的协作方式。

以 OpenClaw 为代表的 human-like agent 这个趋势，对我的启发很直接：当 AI 越来越像一个可以被委派任务、被持续协作、被纳入工作流的执行单元时，我们设计与 AI 的协作机制，也要越来越像在设计人与人的协作机制——要有分工，要有边界，要有 Review，要有验收，也要有对结果负责的人。

问题不再只是怎么把代码写得更快，而是怎么重新分配人与 AI 的工作。

## 结语
AI 编程接下来的变化，可能不只发生在模型参数和 Agent 工程上，也会发生在人与 AI 协作方式的交互设计上。

从这个角度看，我们真正要调整的，已经不只是工具，而是人与 AI 的生产关系。 人在这个过程中，可能会逐渐从执行者转向整个 AI 流程中的决策单元：不再亲自参与每一步实现，而是负责定义目标、分配任务、判断结果，并决定哪些产出被接受、被返工、被整合。

这也是我现在对未来 AI 编程协作范式的理解：
AI 负责执行，人负责组织；AI 负责展开，人负责收束。
