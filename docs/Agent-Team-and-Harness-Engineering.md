# Agent Team 与 Harness Engineering

我想做的，是一个能够自动或半自动组织和执行工作的 Agent Team。

我关心的不是同时跑多少个 Agent，而是系统能不能把一组 Agent 组织成一个持续朝同一目标推进的执行结构。这个问题往下拆，最后会落到四件事上：

- 目标是否先被收敛
- 执行是否有明确分工
- 运行时是否有中心协调
- 整个过程是否可观测、可审查、可复盘

后面的很多设计，其实都是围绕这四件事展开的。

## 一、先把目标收敛成契约

大型任务真正困难的地方，通常不是执行本身，而是目标在一开始并不稳定。需求里会混着目标、边界、约束、假设和验收标准。如果这些东西没有先被收敛，多 Agent 并行只会把模糊放大。

所以我把第一步放在 spec 上。

我关心的 spec，不是一份说明书，而是一份契约。它要在执行前把这轮工作的目标、范围、约束和验收条件固定下来，让后续的任务拆解和执行规划有稳定依据。

这也是我接受 SDD 的原因。对我来说，SDD 最重要的作用，就是要求系统先收敛意图，再进入执行。

## 二、spec 必须同时服务于人和系统

一旦把 spec 看成契约，它就不能只服务于人，也不能只服务于系统。

用户需要阅读、反馈和批准。系统需要消费它，基于它拆任务、生成执行计划、约束运行时行为。这个要求决定了 spec 需要双轨表达：

- Markdown 面向人
- JSON 面向系统

Markdown 保留语义和上下文。JSON 固定机器必须稳定判断的部分。这样 planner、worker 和 observer 围绕的是同一份契约，而不是各自解释同一段 prose。

## 三、Agent Team 需要稳定的角色结构

当我开始从团队的角度理解多 Agent 时，最先稳定下来的，是角色分工。

我现在把整个系统里的角色看成三层：

- Planner
- Workers
- Observer

Planner 负责把需求收敛成 spec，并在用户反馈后继续修订这份契约。  
Workers 负责执行已经批准的计划。  
Observer 负责在执行过程中持续观察环境、协调运行状态，并在必要时直接介入。

这个结构把目标建模、实际执行和运行时协调拆开了。对 Agent Team 来说，这三个能力必须分层，否则系统很快会失去中心。

## 四、编排关心的是创建执行单元

在编排层，我倾向于把每次任务分发都视为一次新的执行单元创建。

原因很直接。编排最关心的是这轮任务需要哪些角色、每个角色的边界是什么、它们应该在什么样的上下文里启动。复用、回收和资源治理当然重要，但那是另外一个模块的问题。

所以我先把假设收紧：分发意味着创建新的 Agent。这样 execution plan 才能保持清晰。

## 五、共享优先，但执行模式由任务事实决定

我倾向共享工作区，因为很多问题本来就存在于共享上下文和共享边界里。完全隔离可以让执行期更安静，但很多问题只是被推迟到了后面。

但这不意味着所有任务都应该共享。执行模式应该由任务事实决定。

- 边界清晰、需要共享上下文的任务，可以共享执行
- 高风险、强耦合、改动面大的任务，可以隔离执行

共享和隔离都只是 execution mode，不是立场。

## 六、冲突治理应该发生在执行过程中

只要共享模式存在，冲突治理就不能放到最后。

我希望系统在执行过程中就感知问题、调整方向、减少覆盖和返工。这也是我接受轻量 claim 的原因。claim 的作用，是把编辑意图和触达范围显式化，为协调提供事实基础。

但我不希望把锁做得很重。对 Agent Team 来说，更重要的不是把底层锁机制推到极致，而是让系统在看到这些信号之后，能够做出合适的运行时判断。

## 七、Observer 是运行时中枢

执行过程中必须有一个中心角色，对当前环境持续负责。

Planner 负责前期契约。Workers 负责局部执行。Observer 负责执行期的整体健康。它要持续观察环境、判断风险、协调冲突、推动汇报，并在必要时调整节奏。

这意味着 Observer 必须具备足够的控制能力。我接受它直接写入 Worker 终端，也接受它在有限范围内触发自动重排。运行时如果没有一个真正可以行动的中心，系统在关键时刻是无法自稳的。

## 八、Observer 的权力必须被看见

Observer 一旦进入控制面，它的行为就必须进入审计链路。

系统需要知道它对谁发了什么指令、为什么这么做、依据了哪些环境事实、造成了什么结果。否则运行时就会变成黑箱，后续的复盘和修正也无从谈起。

所以我接受 Observer 的高指挥权，但这个前提始终成立：它的重要行为必须可观测、可审查、可回放。

## 九、现有观测能力应该被提升为统一事实层

随着设计往前推进，我越来越确认：现有面向人的观测能力，不只是 UI 功能，它们背后已经包含了一部分环境事实。

谁在活跃、谁碰了哪些文件、哪些区域有风险、哪些行为在并行发生、哪些依赖关系会带来影响，这些信息既对人有价值，也对 Observer 有价值。

所以更自然的方向，是把现有观测能力提升为统一的 observation context。人读取它来理解现场，Observer 读取它来做运行时判断。系统里只有一套事实层，后续的协调和审查才容易成立。

## 十、`agents.yaml` 应该回到轻量索引的位置

随着运行时事实越来越丰富，我也越来越清楚 `agents.yaml` 的职责边界。

它适合做轻量索引，承载 pane 和基础 runtime 状态，也继续承担兼容入口的职责。但它不适合继续承担所有环境事实。

所以我倾向于把职责拆开：

- `agents.yaml` 保留轻量索引
- `claims.yaml` 表达公开意图
- `observation.json` 承载聚合后的环境事实
- `observer-commands.jsonl` 承载运行时指挥行为的审计

这个拆分不是为了增加文件，而是为了让每一层都保持清晰。

## 十一、我理解的 harness engineering

我越来越觉得，真正决定 Agent Team 上限的，是 harness。

harness engineering 对我来说，不是某个具体框架，而是一种系统建设方式：把输入固定下来，把契约固定下来，把运行时事实显式化，把关键行为纳入观测和审计，让系统能够复盘、修正、再执行。

它的意义，是给 Agent 的自主性一个边界清晰、行为可见的执行壳。

## 十二、我真正想推进的东西

把这些决策放在一起看，我真正想推进的，是一套面向 Agent Team 的组织系统。

它首先收敛目标，然后形成契约，再生成执行计划，再组织 Worker 和 Observer，在执行期维持秩序，并把关键动作留在可观测和可审查的轨道里。

如果这些事情做成了，自动或半自动组织和执行 Agent Team 才有基础。对我来说，这件事的核心从来不是热闹的并发，而是可治理的协作。
